Portal de autoatendimento do paciente, do zero ao piloto

Hosp360 (nome interno CRMHosp) é a fundação técnica de um sistema de autoatendimento hospitalar: portal Next.js, API NestJS, banco PostgreSQL e Keycloak, todos de pé em minutos via Docker Compose — com um fluxo de agendamento já funcional ponta a ponta.

O que é

Um monorepo de base para o autoatendimento de um hospital

Hosp360 não é um hospital "de mentira" com dados reais — é a base técnica (scaffold) da primeira frente de um sistema maior: o módulo em que o paciente consulta agenda, pede horário e acompanha o status, sem depender da recepção pra tudo.

🖥️ Portal do paciente

Landing institucional, login estrutural, painel, agendamento, consultas, atendimento, notificações e admin — 8 telas em Next.js, já navegáveis.

🔌 API de agendamento

NestJS com endpoints reais de especialidades, horários e solicitações de agendamento, com dados em memória e geração de protocolo.

🐳 Infra local em um comando

Docker Compose sobe PostgreSQL e Keycloak (com seu próprio Postgres) prontos pra integrar autenticação e persistência nas próximas fases.

Como funciona

O fluxo previsto para o paciente

Esta é a jornada que o portal já modela nas telas (parte com dados mock, parte já consumindo a API real de agendamento):

Acessa o portal→ Login / cadastro→ Consulta agenda→ Solicita agendamento→ Acompanha status→ Atendimento via WhatsApp→ Alertas internos via Telegram

🌐 apps/web

Portal do paciente em Next.js — App Router, páginas estáticas (export).

⚙️ apps/api

API de autoatendimento em NestJS — dados in-memory, sem banco ainda.

🗄️ PostgreSQL

Banco operacional via Docker, previsto pra persistência futura.

🔐 Keycloak

Servidor de autenticação de pé no Docker, sem integração real ainda.

Pré-requisitos

O que precisa estar instalado

Ambiente de desenvolvimento local, sem serviços externos obrigatórios.

Node.js 20+

O package.json raiz declara engines.node >= 20.0.0. Workspaces npm (apps/*, packages/*).

# confirma a versão
node -v

Docker + Docker Compose

Sobe PostgreSQL 16 e Keycloak 26.1 localmente — nenhum dos dois precisa ser instalado à parte.

# confirma o Docker
docker compose version

.env local

O repo traz um .env.example com todas as variáveis (Postgres, Keycloak, URLs públicas). Copie antes de subir o Compose.

# cria o .env
cp .env.example .env
Guia de uso · passo a passo

Do clone ao portal rodando local

Comandos reais, na ordem que o README do projeto recomenda.

1

Clone e entre no repo

Monorepo com npm workspaces: apps/web, apps/api, packages/config.

git clone https://github.com/inematds/hosp360.git && cd hosp360  # baixa o projeto
2

Configure o ambiente

Copie o .env.example — ele já traz valores de desenvolvimento para Postgres e Keycloak.

cp .env.example .env  # ajuste as senhas se for além de uso local
3

Suba a infraestrutura local

Docker Compose levanta PostgreSQL (porta 5432) e Keycloak (porta 8080, modo start-dev).

docker compose up -d  # crmhosp-postgres + crmhosp-keycloak-postgres + crmhosp-keycloak
4

Instale as dependências

Um único npm install na raiz resolve os workspaces de apps/web e apps/api.

npm install  # instala web + api + packages/config
5

Rode a API

NestJS em modo watch, porta padrão 3001 (variável API_PORT).

npm run dev:api  # GET /health, /roadmap, /specialties, /slots, POST /appointment-requests
6

Rode o portal

Next.js em modo dev, consumindo a API via NEXT_PUBLIC_API_URL.

npm run dev:web  # http://localhost:3000
7

Navegue pelas telas

Landing, login, painel, agendamento, consultas, atendimento, notificações e admin — todas já existem em apps/web/src/app.

open http://localhost:3000/agendamento  # fluxo piloto que já consome a API real
⚠️ Estado atual: os dados da API ficam em memória (arrays no AppService) e reiniciam a cada restart — ainda não há persistência em banco nem autenticação real via Keycloak. É uma base de fundação técnica (Sprint 1 do todo.md), não um sistema em produção.
